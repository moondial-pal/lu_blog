BASH Scripting: First Automation

#2 - Oct 4, 2020





Since I began learning about linux I have come to realize that I acutally really enjoy it.
I am constantly looking to discover new tools that could help me on my journey, linux or otherwise.
The Bash terminal is "o.g." - one of the original terminals. So I feel like it is good to experience and appreciate.
When I learned about bash scripting I was kind of shocked at how easy it was to pick up.

I must say it was helpful having learned a little python. Concepts such as
variables and functions were not super foreign to me =hit-ground-running.





Above I ran the "pwd" command to show my working directory.

Being in the "/scripts" folder, I am going to create and edit a file with the "vim sysup.sh" command.
If you have never used vim, I suggest you read the docs or watch a turorial for the fundamental/basic commands.




Now to write the script.
Save and quit vim





"#!/bin/bash" This is telling your system to use bash as your command interpeter.

"sudo" assigns you root privlages, needed to execute the next two commands.

"apt-get update" This will update the package list for the upgrades. It does not install anything.

"sudo apt-get upgrade" This will install the the updates and upgrade the packages.

"echo" Displays text. Echos string(s) to standard output, as stated in the manual.

Save and quit vim.

Next to use chmod to initialze the file to execute with "chmod +x sysup.sh"




"chmod" changes the the mode of the file permissions. "+x" is adding it to execute.
Of course "sysup.sh" is the file containing the script.

Now run the script "./sysup.sh"




The pre-fix on the file name "./" means current directory.




There are two sudo commands in the scripts, so the root pass will be requested.
The pass is only asked for once. I like that, it is a good habit when executing root level commands.




Since I ran this script recently I don't have any updates currently.

This was my very first bash script. It got me pretty psyched and I appreciate scripting in general because of it.







Cloudscaping to Landscaping

#1 - Sep 18, 2020



Hello, and welcome!

You may be curious about the title. I am in the landscape business and am essentially transitioning to
predominantly dev/ops and cloud development work.
Navigate to "about" above if you would like to know my story.

This is a very simple web app that I made with Google App Engine and flask w/python. I'd like to give
a brief, general over view of the tools and different methods I used to develop this.
Also, this is my first experience with markdown - its honestly embarrassing. On the up side, I can only
see it getting better.

The main focus of this post is the process.

Tools I used to deploy this web app



    Windows 10 OS - Running on my lenovo laptop. My beloved work horse.

    Sublime Text 3 - To write code.

    Git Bash terminal - I use this to interact between github and my local code and machine.

    Git Hub - Open source, great for version control and for sharing code with other developers

    Google Cloud Platform - GCP is a great platform, the almighty cloud whos services are to be leveraged.

    Google Cloud SDK - A command line interface tool, such that once you get a hang of the gcloud syntax, it makes
    developing with Google Cloud Platform more efficient.

I like using the The GCP console (User Inerface) it's great, but for this project I only popped in to the console just to check up on a few things.

It is very convenient to be able to toggle back and forth.
Let's get started.

Sublime Text 3 has been great to work with so far. I have one or two more text editors on my radar that I will be trying out.
For now Sublime is doing the trick just fine.

First step is to write the code, and to be sure your files are structured properly for the flask frame work.
For instance, flask will specifically look for a folder "templates" and render the file specifed in that folder.
That way when a user clicks the "blog", "contact", or "about" tab there is something there.



Below you can see the folder "templates" and files within, highlighted on the left.





Google App Engine offers two types of environments that the code can be ran in.
Standard and Flexible.
For this very simple project, I am running the code in the standard environment.

Below is what my "standard environment" app.yaml looks like.





Google has great documentation. I was able to confirm how the yaml file is written for the App Engine standard environment.
They give you the rundown quite well.
Notice their comments marked by a hashtag.

I'll also need to specify the version of flask required for the app in my "requirements.txt", also located in my working directory.





Now to open the Google Software Development Kit, aka the Google SDK.
In this case my working directory or the folder
I will be working out of is "'Cloud SDK'". For me this is located here,
C:\Users\User\'Cloud SDK'





Good, now I am going to clone the repository (lu_blog) from my git hub account into my working directory
using the "git clone" command, see below.





I like to confirm by scoping my directory simply with the dir command.
"dir"





Now I am going to change directory into our cloned repo. I'll use the cd command,
"cd lu_blog"





Ok, now to deploy the code to Google App Engine.
"gcloud app deploy"





GCP will then spin up an instance, this will take a minute or two. It may ask to confirm services to deploy.
I'll type "Y" to continue.





App engine will proceed to deploy the web app to the following address https://blog-test-02.wl.r.appspot.com





Now I will type the following to use the app in a browser.
"gcloud app browse"





When you input that command, a browswer window will open up with the web app...The app is up and running and if you are reading
this post, you are looking at an updated version. So this service is super helpful if you want to focus more on the code than the infrastructure.
Google App Engine helps in a big way with that, by splitting traffic, autoscaling and facilitating different methods of deployment.
